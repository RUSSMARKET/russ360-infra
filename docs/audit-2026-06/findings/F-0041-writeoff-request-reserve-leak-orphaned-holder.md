## Симптом
При создании заявки на списание резерв (`qty_reserved += qty`) держится на балансе бессрочно, пока заявка `pending`. У `pending`-заявки нет TTL/авто-отмены. Если держатель выбывает из проекта до решения менеджера, зарезервированное количество остаётся «замороженным» в `qty_reserved`, занижая `available = qty_total - qty_reserved` для всех остальных операций этого держателя.

## Доказательства (file:line)
- `app/Services/Inventory/InventoryWriteoffRequestService.php:48-55` — резерв ставится при `createRequest`.
- Снятие резерва только в `approveRequest` (~`:168`), `rejectRequest` (~`:245`), `cancelRequest` (~`:291`) — все требуют действия менеджера или автора.
- `:269-271` — `cancelRequest` доступен только `requested_by`; если автор недоступен, а менеджер заявку не трогает, резерв не освобождается.
- Нет фонового release / expiry pending-заявок (нет cron/команды, снимающей резерв по таймауту).

## Триггер / repro
Промоутер создаёт writeoff-request (резерв на его балансе), затем увольняется/выходит из проекта до решения менеджера; менеджер заявку не reject'ит → у держателя постоянный фантомный резерв, его реальные остатки недоступны к выдаче/передаче.

## Корневая причина (гипотеза)
Резерв — побочный эффект `createRequest` без компенсирующего механизма освобождения по таймауту или по событию изменения membership держателя.

## Радиус поражения
Краевой; занижение доступного остатка конкретного держателя по конкретным SKU. Накапливается при росте числа брошенных pending-заявок.

## Направление фикса (1-2 строки, НЕ реализовано)
Либо авто-expire pending-заявок (cron + release резерва в той же транзакции), либо разрешить менеджеру force-cancel с освобождением резерва. Обсудить, не реализовывать.
