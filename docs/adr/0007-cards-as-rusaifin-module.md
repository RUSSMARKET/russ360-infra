# ADR-0007: Учёт пластиковых карт — serialized-модуль в rusaifin, rusairegistry отложен

**Status:** accepted
**Date:** 2026-06-11

## Context

Бизнес-боль: банк-партнёр (ОТП) поставляет в офис пластиковые карты коробками (10–20 коробок × 500–700 карт за поставку, дебет/кредит). Карты развозятся по точкам, агенты выдают их клиентам вместе с заявкой в приложении партнёра. Сейчас физические карты **нигде в системе не учитываются** — только в локальных таблицах вне платформы. Не видно: сколько карт на какой точке, какие потери по точке, когда пора пополнять точку, чтобы она не встала.

В roadmap под это был заложен **M4 = rusairegistry** — отдельный 5-й сервис под полную durable-asset модель (AssetType/AssetUnit/Serial/HolderHistory/LossEvent, lifecycle с suspend/resume и перевыдачами; спека `rusaicore/docs/russmarket360/03-assets-registry-rusairegistry.md`).

При проработке выяснилось:
- Карта ОТП — **расходник с односторонним потоком** (офис → точка → клиент, безвозвратно), а не durable-актив с историей держателей и перевыдачами. Модель из спеки registry (holder history, suspend/resume) для неё избыточна.
- Боли («сколько на точке», «потери», «пополнить вовремя») — это поштучный складской учёт по серийнику (штрихкод конверта), сканируемому на каждом шаге.
- Все участники процесса (саппорты, РГ, агенты) работают **в fintech/rusaifin** и на склад (rusaisklad) не заведены.
- Ручник (`ruchnik`) уже фиксирует выдачу заявки агентом, а `ruchnik_reports` уже реализует сверку с внешним отчётом партнёра по коду — половина инфраструктуры существует.
- Durable-активы (планшеты/SIM) как реальная потребность — **подтверждённо отсутствуют**.

## Decision

Учёт карт строим **самодостаточным serialized-модулем внутри rusaifin** (`app/{Models,Services,Http/Controllers,Enum}/Cards/`, свои миграции), а не отдельным сервисом.

Ключевое: ведём **поштучный реестр единиц** (`card_units` по уникальному коду), остаток на точке — **производный** (`COUNT(units WHERE point=X AND state=at_point)`), а не количественный баланс. Это убирает целый класс баланс-рассинхрон багов (ср. F-0034/F-0036/F-0040 в аудите sklad — их физически нет при производном остатке). Из rusaisklad переиспользуем **паттерны** (журнал движений, two-phase передача «отправил→принял», инвентаризация снапшотом, `lockForUpdate` в транзакции), но не таблицы количественных балансов.

Модель: `card_batches` (коробка = батч), `card_units` (карта: code = штрихкод конверта, state, holder = Core-точка, `ruchnik_id` при выдаче), `card_movements` (журнал-аудит потерь), `card_shipments` (партия офис→точка, two-phase, идемпотентность по явному составу кодов). Жизненный цикл: `in_central → in_transit → at_point → issued`, ветви `lost` и `returned`.

**rusairegistry как отдельный сервис — отложен** (не отменён). Спека `03-assets-registry` сохраняется как заготовка ровно на случай, если durable-активы реально появятся.

## Alternatives

- **A. Карты как инвентарь в rusaisklad_back** + S2S-списание из rusaifin. Переиспользует всю складскую механику, но: все пользователи карт живут в fintech (не в sklad) → S2S-прокси на каждый экран или заведение всех во второе приложение; количественные балансы избыточны для serialized-учёта со сканом.
- **C. Отдельный сервис rusairegistry по спеке 03.** Пятый сервис на одного разработчика; durable-asset модель не ложится на расходник-карту; durable-активов нет. Отложено до реальной потребности.
- **Расширить ручник полем «код карты».** Смешивает две разные сущности: `ruchnik.code` = номер заявки в приложении ОТП (внешний), у карты — другой код. Связываем их (`card_units.ruchnik_id`), но не сливаем.

## Consequences

**Получаем:** учёт идёт там, где живёт процесс и пользователи (fintech); переиспользование отработанных паттернов sklad без переписывания; производный остаток исключает баланс-баги; готовая инфраструктура сверки (клон `ruchnik_reports`); бесплатный роллап «по территории» после ADR-0006 (точка получит `territory_id`); скан-сканер как HID-клавиатура — без спецдрайверов.

**Платим:** в rusaifin (legacy-монолит) добавляется новый домен — компенсируем строгой изоляцией модуля (зависит только от `User`/`Ruchnik`/`Point`/Core-gateway), чтобы вынос в сервис был механическим. M4/M5 в roadmap переопределяются.

**Путь извлечения в сервис** (если понадобится): `card_*` таблицы → БД нового сервиса; точки/сотрудники уже за Core-gateway; связь карта↔ручник из FK → S2S/событие `RuchnikCreated`. Модуль спроектирован как «сервис внутри монолита».

## Revisit if

- Появятся durable-активы (планшеты/SIM/сканеры) с реальной потребностью в holder-history → достаём спеку `03-assets-registry`, поднимаем rusairegistry.
- Объём/нагрузка учёта карт начнёт мешать rusaifin → выносим модуль в отдельный сервис по подготовленной границе.
- Карты понадобятся и складу (rusaisklad) как полноценному consumer → ревизия в сторону Core/отдельного сервиса.

До этого — модуль в rusaifin, переоценка не запускается.
