# Russ360 — Architecture Decision Records

Это журнал архитектурных решений по Russ360. Каждый ADR — отдельный файл, фиксирующий **одно** значимое решение: что выбрали, какие альтернативы рассмотрели, какие компромиссы приняли, когда переоценим.

## Зачем

Главное назначение — **анти-паралич-инструмент для solo-разработчика**. Архитектурные сомнения возникают регулярно (статья, разговор, ретроспектива). Без ADR каждый раз приходится прокручивать одни и те же рассуждения с нуля. С ADR — открываешь нужный файл, перечитываешь, идёшь работать дальше.

Вторичное назначение — онбординг будущего разработчика, если он появится. Не «как устроена система» (это в `/docs/russ360-deep-dive.md` и в коде), а **почему она такая**.

## Когда писать новый ADR

Решение значимое, если:

- его отмена потребует > 1 дня работы, **или**
- оно влияет на > 1 сервиса.

Что **не** ADR: имена переменных, локальный code style, разовые баг-фиксы, выбор библиотеки внутри одного сервиса, конкретные процедуры деплоя (это в playbooks).

## Правила ведения

- **Append-only.** Существующие ADR не переписываются, не перенумеровываются.
- **Один ADR — одно решение.** Если решение комплексное — разрезай на несколько ADR.
- **Сменился взгляд?** Создай новый ADR со статусом `accepted`, старому проставь `superseded by ADR-NNNN`.
- **Формат фиксирован** (см. ниже). Не плодим вариации.

## Формат

```markdown
# ADR-NNNN: <одно предложение — что решено>

**Status:** accepted | superseded by ADR-NNNN | deprecated
**Date:** YYYY-MM-DD

## Context
Зачем встал вопрос. Что было до решения. 1-2 коротких абзаца.

## Decision
Одно-два предложения. Что выбрали.

## Alternatives
- **Название** — одна строка почему отклонено.
- (3-5 пунктов)

## Consequences
**Получаем:** короткий список.
**Платим:** короткий список.

## Revisit if
- Конкретное условие 1.
- (2-4 пункта; либо явная пометка «не пересматривается»)

## Links
ADR-XXXX (только если напрямую связаны)
```

Целевой объём — ≤70 строк, читается за 30 секунд.
