# ADR-0001: Deliberate service-oriented architecture

**Status:** accepted
**Date:** 2026-05-21

## Context

Russ360 не проектировался с нуля как многосервисная система. Реальная траектория:

1. Существовали два независимых Laravel-приложения — `rusaifin` (operational sales, MySQL) и `rusaisklad_back` (inventory, PostgreSQL). Свои `users`, своя auth, не связаны.
2. Возникла бизнес-задача — связать учёт ТМЦ с operational приложением (общая идентичность + общие справочники).
3. Из этой задачи родились `rusaiauth` (собственный OIDC IdP) и `rusaicore` (registry для cross-service сущностей).

Итог — 4 backend-сервиса + 2 фронта. Возникает вопрос: это правильная архитектура или мы построили **distributed monolith** (антипаттерн)?

## Decision

Сохраняем 4-сервисную разрезку как **deliberate service-oriented architecture** — осознанный выбор формы под реальные ограничения (один разработчик, два legacy-приложения, необходимость их связать без потери независимости).

«Distributed monolith» — пренебрежительный термин для случая, когда команда страдает от связности **неосознанно**. У нас связность осознана, БД полностью изолированы, контракты явные через REST + JWT, rusaiauth полностью самостоятелен в runtime. Поэтому форма точнее называется **deliberate SOA**, не антипаттерн.

## Alternatives

- **Влить один сервис в другой** — потеря независимой эволюции, большая legacy-миграция.
- **Sync `users` между БД / дублировать справочники** — drift инвариантов гарантирован.
- **Один сервис как owner shared data** — несвойственная ответственность, бутылочное горлышко для будущих сервисов.
- **Внешний IdP** (Keycloak / Auth0) — чужой стек, vendor lock или operational overhead.
- **Полные микросервисы** (service mesh, contract tests, distributed tracing) — требует ≥10 человек только на платформу.

## Consequences

**Получаем:** изолированные БД, security boundary через rusaiauth, готовность core к новым consumers (rusairegistry, LMS), full ownership Laravel-стека без vendor lock.

**Платим:** HTTP overhead на cross-service чтения, связные deploy-окна для cutover'ов, операционная нагрузка на одного разработчика (4 .env, 2 DB engine, несколько git-репо).

## Revisit if

- rusairegistry не появился в roadmap 6+ месяцев после Stage 3 → возможно, core стоит влить в одного из consumers.
- Появилась 2-я команда / 2-й разработчик → возможно, движение в сторону «настоящих» микросервисов.
- p99 межсервисного запроса стабильно > 500 ms под прод-нагрузкой → кэш или слияние сервисов.
- Core начинает содержать бизнес-логику beyond CRUD-зеркала → ревизия границ.

До этого — переоценка не запускается, даже под влиянием внешних статей или разговоров.
