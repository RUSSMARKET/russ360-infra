## Симптом
При применении корректировки `qty_total` уменьшается на отрицательную дельту без проверки инварианта `qty_total >= qty_reserved` внутри транзакции. Защита от конфликта с резервом считается до транзакции на незаблокированном балансе; если между проверкой и применением резерв вырос (новая передача), итог уходит ниже резерва → отрицательный доступный остаток (`available = qty_total - qty_reserved < 0`).

## Доказательства (file:line)
- `app/Services/Inventory/InventoryCheckService.php:256` — `findCorrectionConflicts()` вызывается **до** `DB::transaction` (`:265`).
- `:457-462` — внутри `findCorrectionConflicts` баланс читается **без** `lockForUpdate`.
- `:469-471` — guard `$newTotal < $balance->qty_reserved` существует, но вычисляется на этом нелоченном, потенциально устаревшем балансе.
- `:280-282` — внутри транзакции `$balance->qty_total += $delta; $balance->save();` выполняется **без** повторной проверки против `qty_reserved`.
- Контраст: путь создания движений `app/Services/Inventory/InventoryService.php:1324-1328` блокирует корректировку при `$newTotal < $balance->qty_reserved` непосредственно в момент записи.

## Триггер / repro
1. Инвентаризация: для позиции `qty_actual < qty_expected` (дельта отрицательная), конфликтов нет (`findCorrectionConflicts` на `:256` проходит: `qty_total + delta >= qty_reserved`).
2. До входа в транзакцию `apply-corrections` супервайзер инициирует передачу того же SKU того же держателя → `qty_reserved` растёт.
3. Транзакция `apply-corrections` блокирует баланс на `:274`, применяет отрицательную дельту на `:281` без повторной проверки → `qty_total < qty_reserved`. Доступный к выдаче/передаче остаток отрицателен.

## Корневая причина (гипотеза)
TOCTOU: проверка инварианта `qty_total >= qty_reserved` выполняется вне атомарного контекста и на незаблокированной строке, а сама мутация — внутри транзакции без повторной проверки.

## Радиус поражения
Целостность инварианта `qty_total >= qty_reserved` по конкретным держателям/SKU; последующие передачи/выдачи считают доступным отрицательное число. Требует конкурентной передачи в узком окне между `:256` и `:281`.

## Направление фикса (1-2 строки, НЕ реализовано)
Перенести проверку конфликтов внутрь транзакции на заблокированном балансе либо в цикле применения (`:280`) под `lockForUpdate` проверять `qty_total + delta >= qty_reserved` (как в `InventoryService.php:1324`).
