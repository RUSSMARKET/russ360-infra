# Russ360 — аудит микросервисов (2026-05-18)

Цель: выявить расхождения между декларируемой архитектурой v2 (см. `rusaicore/docs/russmarket360/`) и реальным состоянием кода. Аудит покрывает 6 кодовых баз: `rusaicore`, `rusaiauth`, `rusaifin`, `rusaisklad_back`, `fintech` (front), `rusaisklad_front`.

Формат отчёта: для каждой сущности перечисляются проблемы словами с указанием места (file:line). Без кодовых блоков. В конце — приоритизированный список Top находок и Cross-cutting проблем.

---

## TL;DR

- **P0 (риск безопасности или функциональный)**: Logout на обоих фронтах не ревокирует токены — refresh_token остаётся валидным на rusaiauth после нажатия «Выйти». Cold-start обоих фронтов не валидирует токен на бэкенде. **🟢 ИСПРАВЛЕНО 2026-05-18.** fintech: `User.logoutGlobal` теперь вызывается из адаптера. sklad-front: добавлен `/oauth/revoke` + end_session-flow по образцу fintech; cold-start `session()` теперь дёргает `/auth/me`. Прогон через playwright локально зелёный.
- ~~**P0 (data-drift риск)**: `rusaifin.StaffVisibilityScopeService` читает Memberships двумя путями одновременно — Core gateway и legacy SQL.~~ **🔵 ПЕРЕОЦЕНКА 2026-05-18**: проверка реального кода `StaffVisibilityScopeService.php` показала, что главный метод `computeVisibleUserIds` (строки 85-130) уже идёт **только** через `projectMembershipProvider`. Legacy raw SQL остался в **вспомогательных** методах (`resolveVisibleProjectIdsByUsers`, `filterVisibleIdsByLocalProject`, `resolveAttachedPayload`), где он читает **специфичные для legacy схемы** данные (например, points by group_leader_id), а не дублирует Core. Drift-риска нет. **Понижено до P2** — долговременная чистка читателей, когда у Core будет полный paritet.
- **P1**: Свежая колонка `project_points.core_location_external_id` (заполнена 2026-05-18) сейчас **write-only** — ни один контроллер не читает. **🟡 ПРОДВИНУТО 2026-05-18.** Не подцеплено к читателям, но колонка использована новой командой `core:seed-location-assignments-from-rusaifin` (backfill assignments) и в `PointService` для dual-write. Чтение от контроллеров — отложено до Stage 3.
- **P1**: `rusaifin` контроллеры читают локальные Eloquent (User, Project, Point) напрямую почти везде. Контракты `EmployeeDirectory`, `ProjectCatalog`, `OperationalLocationCatalog`, `ProjectMembershipProvider`, `OperationalLocationAssignmentProvider` зарегистрированы в DI, но используются только в нескольких сервисах (`ProjectBootstrapPayloadFactory`, `CurrentProjectService`, `StaffVisibilityScopeService`).
- **P1**: `rusaifin.User::role_id` читается в 112+ местах. Core Role mapping (`CoreRoleMapper`) определён, но в основной логике не задействован.
- **P1**: `rusaisklad_back` shadow-sync в Core отключён по дефолту (`config/core.php`: `stage1.shadow_sync` = false) — Memberships, Employee, Project пишутся только в локальные таблицы; `ProjectMembershipView.externalId` структурно поддержан, но в реальности всегда `null`.
- **P2**: rusaicore — `project_role` и `assignment_role` хранятся как `text` в БД, не как Postgres enum, несмотря на наличие `ProjectMembershipStatus` enum в коде.
- **P2**: fintech — Reverb-конфиг (`nuxt.config.ts:190-199`) задан, но в коде нет ни одной WebSocket-подписки.

## Детально по сущностям

### Employee

**Canonical: rusaicore.Employee.**

- **rusaicore** — Eloquent `App\Domain\Employee\Employee` (таблица `employees`), генерируемый `external_id` (UUID), nullable `identity_user_id`. Actions работают с Eloquent напрямую (без Repository-паттерна). Status enum — domain enum. Без расхождений.
- **rusaiauth** — не хранит. Связь только через `identity_user_id` UUID в Employee у rusaicore.
- **rusaifin** — `App\Models\User\User` (таблица `users`) с колонкой `core_employee_external_id`. Backfill через `BackfillCoreEmployeeLinksCommand`, `SeedCoreEmployeesFromRusaifinCommand`, `ReconcileCoreDataCommand`. **Проблема**: контракт `EmployeeDirectory` зарегистрирован, но контроллеры читают локальный User Eloquent напрямую (`$user->name`, `$user->role_id` и т.д.); core gateway используется только в `ProjectBootstrapPayloadFactory` и `CurrentProjectService`. **Проблема**: `core_employee_external_id` может быть NULL для legacy users — миграции есть, но полнота не гарантирована.
- **rusaisklad_back** — `App\Models\User` (~38 строк) с `external_user_id` для связи к Core Employee и `identity_user_id` для OAuth. Контракт `EmployeeDirectory` реализован через `LocalEmployeeDirectory`, читает из локальной таблицы. **Проблема**: нет auto-provisioning юзера при первом успешном JWT — если OAuth пришёл с `identity_user_id`, которого нет в `users`, guard вернёт null без понятной ошибки (`app/Providers/AppServiceProvider.php:120-132`).
- **fintech / rusaisklad_front** — только чтение через `/auth/me` (rusaifin) или `/api/v1/auth/me` (rusaisklad). Никакого прямого использования `external_id`.

### Project

**Canonical: rusaicore.Project.**

- **rusaicore** — Eloquent `Project` (таблица `projects`) с `external_id`, `code`, `name`, `status` (domain enum). Actions, без Repository. Route key = `external_id`. Без расхождений.
- **rusaiauth** — не хранит.
- **rusaifin** — `App\Models\Project\Project` (таблица `projects`). **Проблема**: в таблице `projects` **нет колонки `core_project_external_id`** для обратной ссылки — связка делается только через код `RUSAIFIN_{id}` в маппере `SeedProdMirrorFromRusaifinCommand`. Контракт `ProjectCatalog` зарегистрирован и используется в `CurrentProjectService`, `ProjectBootstrapPayloadFactory`, но 49+ контроллеров делают прямые `Project::where()` запросы. **Проблема**: `users.current_project_external_id` добавлена в миграции `2026_04_16_120100`, но в `Project` нет обратной ссылки.
- **rusaisklad_back** — `App\Models\Project` (таблица `projects`) с колонками `core_project_external_id` + `core_project_synced_at`. Route key = `code` без уникального индекса по коду. `LocalProjectCatalog` (контракт `ProjectCatalog`) реализован. **Проблема**: shadow-sync в Core отключён по дефолту (`config/core.php:19`, флаг `stage1.shadow_sync` = false) — `core_project_external_id` не заполняется автоматически.
- **fintech** — читает через rusaifin proxy: `/api/project`, `/api/v1/projects/current`. Список проектов кэшируется в localStorage с ключами `fintech_current_project_id`, `fintech_available_projects`. **Проблема**: точки вложены в `project.points[]` (`src/stores/projects.ts:41-61`) — нет отдельного endpoint, при росте до 1000+ точек структура раздуется.
- **rusaisklad_front** — `fetchProjectsAPI()` (`src/shared/api/projects.ts:98-117`) → `/api/v1/projects` через rusaisklad_back.

### OperationalLocation

**Canonical: rusaicore.OperationalLocation.**

- **rusaicore** — Eloquent с `external_id`, FK на `project_id`, type, status (domain enum). API CRUD на `/v1/operational-locations/{external_id}`. **Минор**: type enum в БД хранится как text, не как domain enum-класс.
- **rusaifin** — `App\Models\Project\Point` (таблица `project_points`). 2026-05-18 добавлена колонка `core_location_external_id` (UUID, char(36) с индексом), backfill прогнан на dev+prod (243/243). **Проблема (P1)**: колонка сейчас **write-only**. `Point::fillable` не содержит её (`app/Models/Project/Point.php:23-36`). Контракт `OperationalLocationCatalog` используется только в smoke-test/reconcile командах, не в основных контроллерах. То есть данные в Core залиты, но переключения читателей нет (Stage 2 не начат).
- **rusaisklad_back** — точки не используются (grep по `OperationalLocation` и `location` пуст). Inventory привязан к супервайзеру/промоутеру через `Assignment`, не к точке.
- **fintech** — данные приходят вложены в `project.points[]` через rusaifin (`projects.ts:41-61`). Pinia-store: `useProjectsStore().points`. Нет отдельного выбора текущей точки в UI.

### ProjectMembership

**Canonical: rusaicore.ProjectMembership.**

- **rusaicore** — Eloquent с FK на employee/project, `project_role` (text), `status` enum, `started_at`/`ended_at`. **Минор**: `project_role` хранится как text, а не как enum (валидируется только в Data-объектах).
- **rusaifin** — **двойное чтение (P0)**. `StaffVisibilityScopeService` (`app/Services/Staff/StaffVisibilityScopeService.php:33-50, :70`) одновременно (а) делает raw SQL в legacy таблицы `project_supports`, `project_regional_directors`, `project_point_agents`, `projects.project_manager_id`, `project_points.group_leader_id` и (б) дёргает `ProjectMembershipProvider::listForEmployeeExternalId()`. Дальше результаты мержатся. Любой write через Core без зеркала в legacy = расхождение в видимости агентов в фильтрах. **Проблема**: `PointService::attachAgent()` и `detachAgent()` помечены `@deprecated`, но активны и пишут в `project_point_agents` без Core-зеркала.
- **rusaisklad_back** — `App\Models\Membership` (таблица `memberships`), pivot User-Project с полем `role` (varchar). `LocalProjectMembershipProvider` реализует контракт. **Проблема**: View.externalId всегда null (`LocalProjectMembershipProvider:59-69`) — структура моста готова, но в данных не заполняется. **Проблема**: `Membership.role` — varchar, не FK на `Role`-enum, нет валидации.

### OperationalLocationAssignment

**Canonical: rusaicore.OperationalLocationAssignment.**

- **rusaicore** — Eloquent с FK на employee/operational_location, `assignment_role` (nullable text), `status` enum. **Минор**: `assignment_role` — text, не enum.
- **rusaifin** — legacy таблица `project_point_agents` (агент↔точка). `OperationalLocationAssignmentProvider` контракт зарегистрирован, есть `CoreOperationalLocationAssignmentGateway` для чтения. **Проблема (P1)**: `CoreOperationalLocationAssignmentWriteGateway` определён и зарегистрирован, но не используется ни в одном контроллере — все писи идут в `project_point_agents` через `PointService` (deprecated, но активен). Read-gateway тоже используется не везде: `StaffService` (`app/Services/Staff/StaffService.php:40+`) дёргает `project_point_agents` напрямую.
- **rusaisklad_back** — `App\Models\Assignment` (таблица `assignments`) — это иерархия supervisor→promoter (не точка→агент). По домену это разное, но **проблема**: нет soft-delete на assignments → при удалении promoter ассигнмент становится orphan.

### IdentityUser

**Canonical: rusaiauth.IdentityUser.**

- **rusaiauth** — таблица `identity_users` (UUID PK + уникальный `external_id`). Прямые Eloquent-запросы в `RegistrationController`, `LoginController`, `UserController`. Нет Domain-Service слоя над моделью.
- **rusaicore** — `Employee.identity_user_id` (UUID) хранится как FK без активной валидации; при удалении IdentityUser в rusaiauth `identity_user_id` остаётся как orphan.
- **rusaifin / rusaisklad_back** — `users.identity_user_id` (UUID), используется в guard'е для резолва. **Проблема**: в `rusaifin` колонка `identity_user_id` не задокументирована в CLAUDE.md/M2 доках (`app/Providers/AppProvider.php:108`).

### OAuth Clients / OAuth Tokens

**Canonical: rusaiauth (Passport).**

- **rusaiauth** — стандартные `oauth_clients`, `oauth_access_tokens`, `oauth_refresh_tokens`, `oauth_auth_codes` (mig `2026_05_04_130000` сделала `user_id → UUID`). `IssuerAwareAccessToken` подключён в `AppServiceProvider::configurePassport()` (`app/Providers/AppServiceProvider.php:209`) — гарантирует `iss` в JWT. `LocalSelfIssuedTokenProvider` минтит S2S JWT без HTTP round-trip (избегает single-worker deadlock). **Минор**: комментарий в `LocalSelfIssuedTokenProvider:23-24` говорит «sub=null для client_credentials», а фактически в `IssuerAwareAccessToken::toString()` для S2S `sub = $client->getIdentifier()`, потому что user-null. Resource-server валидаторы (rusaicore `OAuthTokenValidator:126`) специально нормализуют `sub===aud → sub=null`. Логика верная, но коммент в исходнике вводит в заблуждение.
- Остальные сервисы — resource servers, не хранят токены.

### OTP / SMS verification

- **rusaiauth** — `OtpService` (`app/Domain/Identity/Services/OtpService.php`) с двумя путями: SMS (через `MtsSmsSender`, активируется только если есть `mts.*` config и env != `local/testing`) и email (через `LaravelMailSender`, активируется при non-null mail driver и env != `testing`). На local/testing — Null senders (логирует). На проде memory упоминала P0 баг «не шлёт SMS» — он был починен перед cutover'ом и сейчас не воспроизводится.
- **fintech** — вызовы регистрации напрямую в rusaiauth (`/api/v1/registration/phone`, `/api/v1/registration/code`) через отдельный axios instance `authHttp`.
- **rusaisklad_front** — OTP не используется в фронте (есть только recovery-flow, который тоже идёт в rusaiauth).

### Registration

- **rusaiauth** — публичная регистрация (M2.1): GET `/v1/registration/phone/{phone}`, POST `/v1/registration/phone`, POST `/v1/registration/code`, POST `/v1/registration`. S2S вызывает rusaifin для шага `set-password`. **Минор**: нет Domain Service слоя, всё в контроллерах с прямыми DB-операциями.
- **rusaifin** — `App\Models\Registration\Registration` (таблица `registration`) **ещё живёт**. Internal S2S endpoints (`routes/api.php:54-65`): `/internal/v1/registration/check`, `/init`, `/set-password`. **Проблема**: `RestorePasswordController` (`app/Http/Controllers/Registration/RestorePasswordController.php`) помечен под M2.2 переезд в rusaiauth, но всё ещё активен. **Проблема**: `RegistrationInternalController` и `SetPasswordController` делают direct `DB::table('users')->update()` вместо Eloquent (`app/Http/Controllers/Internal/Users/SetPasswordController.php:13`) — bypass Eloquent observers.
- **fintech** — registration через `/api/v1/registration/*` в rusaiauth, отдельный axios instance `authHttp` (`src/pages/registration/api/index.ts:23-33`). **Проблема**: у `authHttp` нет общих перехватчиков (4xx error mapping, 401 refresh), только у основного `http` (`src/shared/api/http.ts:76-215`) — ошибки регистрации не обрабатываются единообразно.

### Role

- **rusaiauth** — Identity global roles (`global_roles`, mapping через `LegacyRoleMapper`). Backfill-команда `identity:backfill-from-legacy` готова, но неясно, запускалась ли на prod (нет следов запуска).
- **rusaifin** — `App\Models\Role` (таблица `roles`) + `RoleEnum`. **Проблема (P1)**: **112+ мест в коде** читают `$user->role_id` напрямую (контроллеры, сервисы, скоупы). `CoreRoleMapper` определён (`app/Domain/Core/Support/CoreRoleMapper.php`), но используется только в seed-командах для маппинга в Core `ProjectMembership.project_role` — в runtime-логике контроллеров его нет.
- **rusaisklad_back** — `App\Models\Role` (таблица `roles`) + `User.role_id` (global) + `Membership.role` (varchar per project). `User::getRoleCode` (`app/Models/User.php:83-105`) делает fallback project→global→default — это смешивание уровней может давать неконсистентные authz-решения после смены проекта.
- **fintech / rusaisklad_front** — читают `role` из `/auth/me`, кэшируют в `User` объекте/secureStorage. Granular project-level permissions не поддерживаются (см. P1 ниже).

### Auth guard

- **rusaicore** — guard возвращает `OAuthPrincipal` (`sub`, `scopes`). Контроллеры работают с principal. Без расхождений.
- **rusaifin** — **по дизайну** guard возвращает `User` Eloquent (`app/Providers/AppProvider.php:91-124`), потому что ~200 контроллеров читают `auth()->user()->role_id`, `->name` и т.п. (memory `m2_prod_cutover_done`). Это решение зафиксировано как осознанное, но создаёт нагрузку: контроллеры не знают, что аутентификация прошла через OAuth.
- **rusaisklad_back** — то же поведение: guard возвращает полный User Eloquent (`app/Providers/AppServiceProvider.php:103-137`).

### Sanctum / api_token (legacy)

- **rusaicore / rusaiauth** — никогда не было / полностью выпилено.

## Cross-cutting проблемы

### 1. Logout не ревокирует OAuth токены на IdP (оба фронта) — P0

**fintech**: `User.logout()` (`src/entities/user/model.ts:296-298`) вызывает только `logoutLocal()`, который чистит localStorage. `performGlobalOidcLogout()` (`src/shared/lib/oidcLogout.ts:105-134`) реально умеет дёрнуть `/oauth/revoke` для access+refresh и `/oauth/logout` с `id_token_hint`, но **из UI-flow никто его не зовёт**. OIDC adapter (`src/shared/auth/adapters/oidc.ts:66-69`) вызывает только `User.logout()`.

**rusaisklad_front**: симметричная проблема. `oidcAuthAdapter.logout()` (`src/shared/auth/adapters/oidc.ts:110-112`) делает только `User.logoutLocal()`. `performGlobalOidcLogout()` есть (`src/shared/lib/oidcLogout.ts:55-69`), но не задействован в UI.

**Последствие**: после нажатия «Выйти» refresh_token остаётся валидным на rusaiauth до истечения TTL. Если токен утёк (XSS, физический доступ к устройству, sync с другого браузера) — злоумышленник может выпускать новые access_token'ы. Это безопасностный долг, не функциональный.

**Где чинить**: `oidcAuthAdapter.logout()` в обоих фронтах должен вызвать `performGlobalOidcLogout(token, idToken)` ДО `User.logoutLocal()`.

### 2. Cold-start без валидации токена на бэкенде (оба фронта) — P1

**fintech**: `http.ts:14-54` восстанавливает токен из localStorage при старте, и фронт считает пользователя аутентифицированным сразу. Никакого запроса в /auth/me до первого API-запроса не делается.

**rusaisklad_front**: `useSessionStore().bootstrap()` (`src/shared/auth/composables/useSessionStore.ts:175-191`) читает токен из secureStorage и заходит «тихо». Если токен украден/revoked, фронт узнает только на первом API-запросе.

**Где чинить**: добавить вызов `/auth/me` (или `userinfo`) на bootstrap в обоих фронтах, прежде чем поднять UI.

### 3. Двойное чтение Memberships в rusaifin — P0 (drift)

`StaffVisibilityScopeService.php:33-50` делает raw SQL в `project_supports`, `project_regional_directors`, `project_point_agents`, плюс `projects.project_manager_id` и `project_points.group_leader_id`. На строке 70 в этом же файле дёргает `ProjectMembershipProvider::listForEmployeeExternalId()` (Core). Результаты мержатся.

**Последствие**: если write через Core пройдёт без зеркала в legacy таблицы — видимость агента в фильтрах StaffVisibilityScope будет рассогласована с реальностью в Core. Это уже происходит в `PointService::attachAgent/detachAgent` (deprecated, пишет ТОЛЬКО в legacy).

**Где чинить**: после Stage 2 переключения читателей — выпилить legacy ветку из `StaffVisibilityScopeService`. До этого — следить, чтобы все writes шли парой (legacy + Core), либо через единый Application Action.

### 4. core_*_external_id колонки в rusaifin — write-only — P1

- `users.core_employee_external_id` — заполнено бэкфиллом, но в контроллерах читают `User->id` и `User->role_id`, не `core_employee_external_id`. Используется только в `ProjectBootstrapPayloadFactory` и `CurrentProjectService`.
- `project_points.core_location_external_id` — заполнено бэкфиллом 2026-05-18. Не читается нигде. `Point::fillable` не содержит колонку.

**Где чинить**: добавить чтение через `OperationalLocationCatalog` контракт в основных контроллерах (`PointController`, `StaffService`). После — выпилить legacy чтения из `project_points`. Это и есть Stage 2.

### 5. Сontracts orphan в rusaifin — P1

- `CoreOperationalLocationAssignmentWriteGateway` — зарегистрирован, но не используется.
- `EmployeeDirectory`, `ProjectCatalog`, `OperationalLocationCatalog`, `ProjectMembershipProvider`, `OperationalLocationAssignmentProvider` — зарегистрированы все 5 (`app/Providers/AppProvider.php:60-64`), но используются только в `ProjectBootstrapPayloadFactory`, `CurrentProjectService`, `StaffVisibilityScopeService`. ~43 контроллера читают локальные Eloquent напрямую.

### 6. Shadow-sync в rusaisklad отключён по дефолту — P1

`config/core.php:19` — `stage1.shadow_sync` = false. `ProjectMembershipView.externalId` в `LocalProjectMembershipProvider:59-69` всегда выдаёт null. То есть мост к Core структурно готов, но в данных пусто. Если завтра решим сделать sklad-данные видимыми в Core — потребуется backfill (аналог того, что мы сделали для rusaifin 2026-05-18).

### 7. Direct cross-service HTTP — где допустимо

- **rusaicore** ходит ТОЛЬКО в rusaiauth для JWKS-fetch. ✅ канон.
- **rusaifin** ходит в rusaicore через `CoreApiClient` (S2S OAuth client_credentials). В rusaiauth — только JWKS. В rusaisklad — нет. ✅ канон.
- **rusaisklad_back** ходит в rusaicore через `CoreApiClient` (только shadow-sync). В rusaiauth — JWKS. В rusaifin — нет. ✅ канон.
- **rusaiauth** ходит в rusaifin через `RusaifinInternalClient` (S2S для registration flow). Контролируемый исходящий вызов, ✅ канон.
- **fintech** ходит в rusaifin (основной API) и в rusaiauth (только registration + OIDC endpoints). ✅ канон.
- **rusaisklad_front** ходит в rusaisklad_back и в rusaiauth (recovery + OIDC). ✅ канон.

Все cross-service вызовы соответствуют декларации. Прямых вызовов «фронт→rusaicore» нет ни у одного фронта.

### 8. Реверс-связь Project → Employee — P2

- `users.current_project_external_id` (rusaifin) указывает на проект, но в `projects` нет обратной ссылки.
- `Employee` в core знает `external_id` своего проекта через `ProjectMembership`, но в фронте нет API, который вернул бы «список agents этого проекта» через Core — фронт получает их вложенными в `project.points[].agents[]` (это legacy путь через rusaifin).
