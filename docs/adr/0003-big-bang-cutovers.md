# ADR-0003: Big-bang cutovers вместо feature flags

**Status:** accepted
**Date:** 2026-05-21

## Context

При миграциях между схемами / архитектурами (auth-стек, доменные модели, контракты сервисов) есть выбор:

- **Feature flags + gradual rollout** — два контура живут параллельно, постепенный переключатель.
- **Big-bang cutover** — режем в одном окне, до окна один контур, после — другой, без перехлёста.

Feature flags требуют поддержки **двух кодопутей одновременно** (код, тесты, мониторинг, документация). Solo-разработчик не может долго держать two-way compatibility — это утроение поверхности багов и потеря темпа на бизнес-задачах.

## Decision

Используем **big-bang cutovers** — режем в одном окне без feature flags. Pre-cutover smoke и dumps обязательны, rollback план готовится заранее. После окна остаётся один путь.

## Alternatives

- **Feature flags + gradual rollout** — нет мощности поддерживать два контура месяцами.
- **Blue-green deployment** — требует инфры (LB, дубли БД), для одного сервера overhead.
- **Strangler fig pattern** — растягивает миграцию на месяцы, увеличивает поверхность багов.

## Consequences

**Получаем:** чистый код после cutover (один путь, нет dead branches), короткое окно риска (часы, не месяцы), нет долгого maintenance двух кодопутей, простота тестирования.

**Платим:** концентрированный риск в окне (если что-то пошло не так — все юзеры affected сразу), обязательная подготовка (dumps, rollback процедура, smoke checklist), невозможна частичная миграция отдельных юзеров, downtime в окне (обычно минуты).

## Revisit if

- Команда выросла до 2+ разработчиков — поддержка feature flags становится дешевле.
- Появилось бизнес-требование «no downtime / zero-impact migration».
- Cutover risk оценивается как multi-day high-impact (например, окно > 6 часов).
